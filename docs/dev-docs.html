Field Training Scheduling System 3

Developer's Guide

FTSS is a client-side Model View Controller (MVC) web application. The app is written in JavaScript (JS) using Google's AnguarJS framework. All backend code travels through an open-source library, angular-sp, to an eis.af.mil SharePoint (SP) 2010 instance. Security is handled through SP's native interface. The angular-sp library creates Representation State Transfer (REST) queries that are Microsoft Open Data Protocal (oData) 2.0 compliant. This library also builds Atomicity, Consistency, Isolation, Durability (ACID) compliant batch transactions as well as transparent caching and JS Object Notation (JSON)/datetime conversions.

FTSS 3 also makes heavy use of HTML 5 and CSS 3 features that require a modern browser. To deal with this limtitation, browser detection is incorporated which redirects users of unsupported browsers to an update page with instructions on how to continue. All HTML is written in JADE, a terse templating language for NodeJS. The use of Jade mixins at build time allow the creation of very complex HTML and behaviors with little code. Finally, icomoon.io is used to manage and compile the applications icon library into a single Scalable Vector Graphics (SVG) file. SVG was chosen for it's flexibility (CSS-rule inheritance and lossless scaling) and because it was not blocked by standard AF policy. Web fonts are a popular alternative but are blocked in IE on AFNET.

In the event of catastrophic dataloss requiring the complete replacement of the SP instance, the FTSS.wsp file holds the entire SP solution and base data needed to create a new copy of FTSS. Once loaded, the source should be updated to point to the new SP instance and code re-compiled using Brunch.

File Structure

This section details overall file structure and highlights some import files. You can click to expand or collapse folders.

  • _public This folder holds the application build and is not synced in GitHub.
  • app This is the main app source code (no library code).
    • assets Stores any static content that should be copied to _public during build. Right now it only holds our migration page.
    • controllers Holds global controllers that are shared across the entire app.
      • ftss-controllerHelper.js This is main controller code for all views. Every view instaniates this controller for data-binding.
      • ftss-main-controller.js This controller is called during app initialization and builds the parent scope for the whole app.
      • ftss-support-controller.js Controller used for our view-based Q&A feature. This controller includes a polling feature to check for updates continuously for a chat-like user experience.
    • directives Stores the JS for our AngularJS directives.
    • partials The majority of our view or feature-specific view and controller code. Organized by view/feature.
      • directives This folder contains any directives-specific views.
      • includes This folder contains view code that is only used by Jade during build time (from includes).
      • __mixins.jade A collection of powerful Jade functions that are only run once (at build-time).
    • scripts Holds JS shared across the entire app but not a directive or controller.
      • angular-sharepoint.js The AngularSharepoint library that performs all the REST interaction between SP and Angular.
      • dodconsent.js Handles our mandatory DoD consent message and sets the cookie so SP doesn't freak out.
      • ftss-angular-filters.js Custom Angular filters for use with ng-repeat collections.
      • ftss-filters.js This handles our SP tag filtering; it converts from tags to SP lookups.
      • ftss-init.js Performs the app JS initialization and establishes the Angular routes.
      • ftss-jquery-bindings.js We use jquery live bindings because they just make more sense than directives in some cases.
      • ftss-models.js Defines all our SP models for AngularSharepoint.
      • ftss-utils.js Various helper functions used throughout the application
    • styles Holds all the CSS that does not belong to a specific view.
    • index.jade This converts to index.html and is our main landing page that starts the app.
  • bower_components This holds all the managed library code (managed with Bower).
  • docs This documentation source and compiled code.
  • node_modules All the NodeJS apps that are needed by Brunch.io to perform our builds, they are not synced on Github.
  • scripts Development environment scripts, we use server.* for development and production.* for production builds.
  • test All AngularJS tests are here (will be moved into partials later on).
  • vendor Any special or custom library code or other JS that must execute before anything else goes here.

Development Environment Setup

This section covers the recommended development environment setup and guides you through everything you need to get up and running with the FTSS development, testing and build process. This section assumes you have a basic understanding of client-side model, view, controller applications.

The production version of FTSS has a very small index.html file that handles legacy IE detection/redirection, loads our JS/CSS and initializes AngularJS. In the production build there are only three JS files and one CSS file:

  • app.css - All compressed stylesheets.
  • vendor.js - All JS compressed libraries.
  • partials.js - The compiled views (compiled into HTML views from Jade).
  • app.js - All our JS from the app folder compressed into one file.

We use Brunch.io for building both test and production versions of FTSS. Brunch is a NodeJS app and relies on on a three important config files to operate:

  • config.coffee - a CoffeeScript file, contains all the important instructions that allow Brunch to automate our build processes.
  • package.json - defines all Brunch plugins (uses the Node Package Manager).
  • bower.json - defines all our dependencies with version for FTSS.

These three files define the build process and are use to keep our dependencies always up-to-date. Since Brunch does auto updates our dependencies during build (if the version in bower.json permits it), we keep the bower_components directory in our GIT repository to track any breaking changes by external developers. This is also necessary as some libraries have some customizations for FTSS.


Follow the steps below to get up and running now. You can also click the videos at the bottom which walk you through these steps.

  1. Install NodeJS and Node Package Manager (NPM) on your development computer: howtonode.org.

  2. Create or login to GitHub and install the GitHub app: help.github.com/articles/set-up-git.

  3. Clone the FTSS repository: github.com/jeff-mccoy/FTSS.

  4. Open a terminal or command prompt and change to the cloned repo directory on your local computer.

  5. Run npm install to install all the Brunch.io plugins and dependencies.

  6. Run scripts/server.bat or scripts/server.sh to build a development version of FTSS that auto-updates.

  7. Open your browser to localhost:3333 with Cross Site Scripting (XSS) enabled.

     If you use Google Chrome (the recommended development browser), simply run Chrome from a terminal with the
 following command: Chrome --args --disable-web-security.
